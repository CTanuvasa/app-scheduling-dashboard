# APS Scheduling Dashboard

Live coverage planning dashboard for All Pro Security (Region 2) built on the TrackTik API.
The static HTML is rebuilt every 10 minutes by GitHub Actions and published to GitHub Pages.

## What it does

- **Weekly Breakdown** - officers under 30 hrs grouped by 0-8 / 8-16 / 16-30 buckets
- **Monthly Tracker** - officers under 40 hrs with red "At Risk" flag for anyone under
  the 24-hour monthly policy minimum, plus contact-count and last-contact columns
- **Call Off Aid** - pick a site, see eligible officers (filtered by position-skill
  requirements) sorted by distance, with ETA, shift timing, driving permissions, and
  inline contact-log buttons
- **Extra Help Request** - geocoded address/city search with Unarmed / Armed / Driving
  filters; supports future date/time for forward planning

Contact log lives in browser `localStorage`. Cooldowns deprioritize officers automatically
(Yes = 12 h, No = 8 h, No Answer = 30 min, Maybe = 15 min).

## One-time setup

### 1. Create a new repo on GitHub
Push the contents of this folder *minus* the local-only files:

```
scheduling-server.py
logo.png
.github/workflows/refresh.yml
.gitignore
README.md
```

Do **not** push `Setup Credentials.command` (contains keys), `Start Dashboard.command`
(local launcher), or `scheduling-dashboard.html` (regenerated by CI).

### 2. Add the TrackTik secrets
In the repo: **Settings -> Secrets and variables -> Actions -> New repository secret**.
Add all five, exactly:

| Name                   | Value                                     |
|------------------------|-------------------------------------------|
| `TT_TENANT`            | `apsguards`                               |
| `TT_API_CLIENT_ID`     | (from TrackTik Admin -> API Service Acct) |
| `TT_API_CLIENT_SECRET` | "                                         |
| `TT_API_USERNAME`      | "                                         |
| `TT_API_PASSWORD`      | "                                         |

Secrets are encrypted at rest. They are never displayed back, never logged, and the
Python script reads them via `os.environ.get(...)` inside the runner.

### 3. Enable GitHub Pages
**Settings -> Pages -> Source: GitHub Actions** (not "Deploy from a branch").
The workflow's `deploy-pages` step handles the rest.

### 4. Trigger the first run
**Actions -> Refresh & Deploy Dashboard -> Run workflow**. Subsequent runs happen
automatically every 10 minutes plus on any push to `main`.

When the workflow finishes, your dashboard URL is shown on the workflow page (something
like `https://YOUR-USERNAME.github.io/REPO-NAME/`). Bookmark it; share with schedulers.

## Cost / plan notes

- **Public repo + GitHub Free**: unlimited Actions minutes, Pages site is publicly
  reachable (the URL alone is the only barrier).
- **Private repo + GitHub Free**: 2,000 Actions minutes/month - a 10-minute cron uses
  ~8,000+ minutes/month. Either pay (Pro is $4/user/mo, 3,000 min) or stretch the
  cron to `*/30 * * * *`.
- **Private repo + GitHub Team**: needed for private Pages (limits Page visitors to
  org members) - the cleanest auth setup.
- **Public repo + Cloudflare Access**: keep the repo private if you like, deploy the
  built site to Cloudflare Pages instead, and gate it with free Cloudflare Access.
  See `Alternative hosting` below.

## Running locally for development

Keep using the existing local workflow:

1. Double-click `Setup Credentials.command` once (creates `~/.tracktik/credentials`).
2. Double-click `Start Dashboard.command`. Server starts at <http://localhost:8765>.

The Python script checks env vars first, then falls back to `~/.tracktik/credentials`,
so the same code runs in both contexts unchanged.

## Files

- `scheduling-server.py` - canonical script; pulls TrackTik, renders HTML, optionally
  serves it via a stdlib HTTP server. Run `python3 scheduling-server.py --build-static`
  to produce a single `scheduling-dashboard.html`.
- `logo.png` - APS shield (CMYK PNG), base64-embedded into the rendered HTML.
- `.github/workflows/refresh.yml` - the 10-min cron + Pages deploy workflow.
- `Start Dashboard.command` / `Setup Credentials.command` - local-only launchers
  (not in the repo).

## Security notes

- Credentials live in **GitHub Secrets**, encrypted at rest with libsodium sealed boxes
  and auto-masked from workflow logs. They never appear in committed files or in
  generated HTML.
- The generated HTML contains employee names, home cities, weekly hour totals, and
  client site names. Treat the Pages URL as sensitive even though it's "static".
  If your repo / Pages site is public, anyone with the URL can read it.
- Rotate the TrackTik service account credentials any time they may have been exposed
  (chat, screenshots, ex-employees, etc.). Issue from TrackTik **Admin -> Integration
  -> API Service Accounts**, then update the five Secrets in GitHub.

## Alternative hosting (free private)

If you'd rather not pay for GitHub Team but still want auth-gated access:

1. Keep the repo private (Free plan is fine for one developer).
2. Replace the `actions/deploy-pages` step with a Cloudflare Pages deploy step.
3. Put **Cloudflare Access** in front - free for up to 50 users, gates the site
   behind Google/Microsoft/email-OTP login.

Same Python, same Secrets, same refresh cadence; just different deploy target.
